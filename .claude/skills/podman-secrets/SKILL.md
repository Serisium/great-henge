# Podman secrets for VPS services

Every credential a containerized service needs at runtime is stored as a
**rootless podman secret** owned by the service account (`pangolin`, uid 2000).
Secret values must never exist in the repo, the image, or as plaintext files
under `/var/lib` — the repo and image are public.

## Generate on first boot

Secrets are created by a root oneshot `*-config-init.service` that shells out
to the rootless user. Reference implementation:
`vps/usr/libexec/authentik/config-init.sh` +
`vps/etc/systemd/system/authentik-config-init.service`.

The pattern:

```sh
AS_PANGOLIN="runuser -u pangolin -- env XDG_RUNTIME_DIR=/run/user/2000"

# logind hasn't created /run/user/2000 yet at this point in boot
install -d -o pangolin -g pangolin -m 0700 /run/user/2000

if ! $AS_PANGOLIN podman secret exists my-secret; then
    openssl rand -hex 32 | tr -d "\n" | $AS_PANGOLIN podman secret create my-secret -
fi
```

- Generate values with `openssl rand -hex 32`; pipe via stdin (`create ... -`),
  never pass the value as an argument or write it to a temp file.
- **Always strip the trailing newline** (`tr -d "\n"`): podman stores stdin
  verbatim, and a newline inside the secret silently corrupts anything built
  from it — e.g. Go rejects HTTP Authorization headers containing one
  (authentik's embedded outpost died exactly this way).
- Make the script idempotent (`podman secret exists`) and gate the service
  with `ConditionPathExists=!<marker>` so it is a no-op after first boot.
- Order the user manager after the init service with a drop-in at
  `vps/etc/systemd/system/user@2000.service.d/` (see `20-authentik.conf`) so
  the quadlets never start before their secrets exist.
- Secrets persist in the user's podman storage under
  `/var/home/pangolin/.local/share/containers/` (on `/var`, so they survive
  `bootc upgrade`).

## Reference from quadlets

In the `.container` file, inject as an env var (most services) or as a file
at `/run/secrets/<name>` (drop the `type=env`):

```ini
[Container]
Secret=authentik-pg-password,type=env,target=POSTGRES_PASSWORD
```

## Externally-issued secrets

Secrets that can't be generated (e.g. an API token issued by a third party)
are seeded manually on the host, never committed — either
`podman secret create` as the service user during provisioning, or a
root-owned env file like `/etc/tailscale/authkey.env` (see `vps/README.md`).

## Exceptions

Pangolin predates this convention: its `server.secret` is generated by
`pangolin-config-init.service` into `/var/lib/pangolin/config/config.yml`
(0600) because Pangolin reads it from its config file. Don't copy that pattern
for new services, and don't migrate Pangolin's secret without a plan for the
live host (the value must not change).

## Inspecting on the VPS

```sh
sudo -u pangolin XDG_RUNTIME_DIR=/run/user/2000 podman secret ls
```
